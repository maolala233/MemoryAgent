Core Data Structures Reference
================================

This document is the complete reference for Mandol's core data structures, covering all attributes, methods, and interface signatures. For a quick concept overview, see :doc:`/basic-user/basic-concepts`; for terminology mapping, see :doc:`/shared/glossary`.

.. _dsref-memory-unit:

MemoryUnit
-----------

``MemoryUnit`` is the smallest storage unit in the memory system. Each unit contains raw data, metadata, and optional vector representations.

Core Attributes
^^^^^^^^^^^^^^^

.. code-block:: python

   from mandol import MemoryUnit, Uid

   unit = MemoryUnit(
       uid=Uid("dialogue_001"),
       raw_data={
           "text_content": "Zhang San went to Beijing on a business trip",
           "speaker": "user",
       },
       metadata={
           "timestamp": "2024-01-15T10:00:00",
           "spaces": ["root_base_memory"],
           "session_id": "session_001",
       },
       embedding=None,
       sparse_embedding=None,
   )

.. list-table::
   :header-rows: 1
   :widths: 20 30 50

   * - Attribute
     - Type
     - Description
   * - ``uid``
     - ``Uid`` (defined in ``mandol/domain/types.py``)
     - Unique identifier
   * - ``raw_data``
     - ``Dict[str, Any]``
     - Raw data dictionary; system auto-vectorizes ``text_content`` and ``image_path`` fields
   * - ``metadata``
     - ``Dict[str, Any]``
     - System metadata, including timestamps, space info, etc.
   * - ``embedding``
     - ``Optional[numpy.ndarray]``
     - Dense vector representation (generated by EmbeddingProvider)
   * - ``sparse_embedding``
     - ``Optional[numpy.ndarray]``
     - Sparse vector representation (generated by SparseIndex)

Key Methods
^^^^^^^^^^^

.. list-table::
   :header-rows: 1
   :widths: 25 20 55

   * - Method
     - Type
     - Description
   * - ``to_dict()``
     - Instance
     - Serialize to dictionary for persistence
   * - ``from_dict(data)``
     - Class method
     - Deserialize from dictionary to MemoryUnit
   * - ``get_user_metadata()``
     - Instance
     - Get user-defined metadata (excludes system fields with ``_system_`` prefix)
   * - ``touch()``
     - Instance
     - Update ``_system_updated_at`` timestamp

Usage Example
^^^^^^^^^^^^^

.. note::

   The examples below show how to create **basic dialogue units**, added via ``system.add(unit)``.
   Multi-perspective memory units (entities, events, summaries, insights, etc.) are automatically built by the system during ``build_high_level()``,
   and should **not** be manually created or inserted, as this may cause UID conflicts or relationship inconsistencies with system-built units.

.. code-block:: python

   from mandol import MemorySystem, MemoryUnit, Uid

   system = MemorySystem()

   unit = MemoryUnit(
       uid=Uid("dialogue_001"),
       raw_data={"text_content": "Zhang San went to Beijing on a business trip", "speaker": "user"},
       metadata={"timestamp": "2024-01-15T10:00:00"},
   )
   system.add(unit)

   data = unit.to_dict()
   restored_unit = MemoryUnit.from_dict(data)

For the complete ``raw_data`` field reference, see :doc:`/shared/data-format-guide`.

.. _dsref-memory-space:

MemorySpace
------------

``MemorySpace`` is a logical space for organizing MemoryUnits, supporting tree-like hierarchical relationships. Each space can contain multiple memory units and multiple child spaces.

Core Attributes
^^^^^^^^^^^^^^^

.. list-table::
   :header-rows: 1
   :widths: 22 30 48

   * - Attribute
     - Type
     - Description
   * - ``name``
     - ``SpaceName``
     - Space name
   * - ``unit_uids``
     - ``Set[Uid]``
     - Set of contained memory unit UIDs
   * - ``child_spaces``
     - ``Set[SpaceName]``
     - Set of child space names
   * - ``summary_text``
     - ``Optional[str]``
     - Space summary text
   * - ``summary_embedding``
     - ``Optional[numpy.ndarray]``
     - Summary vector
   * - ``metadata``
     - ``Dict[str, Any]``
     - Metadata dictionary

Key Methods
^^^^^^^^^^^

.. list-table::
   :header-rows: 1
   :widths: 35 20 45

   * - Method
     - Type
     - Description
   * - ``add_unit(uid)``
     - Instance
     - Add memory unit to space
   * - ``remove_unit(uid)``
     - Instance
     - Remove memory unit from space
   * - ``add_child_space(name)``
     - Instance
     - Add child space
   * - ``remove_child_space(name)``
     - Instance
     - Remove child space
   * - ``set_summary(text, embedding)``
     - Instance
     - Set space summary and vector
   * - ``get_all_unit_uids(recursive=True, resolver=...)``
     - Instance
     - Recursively get all unit UIDs (requires resolver for recursion)
   * - ``get_all_child_space_names(recursive=True, resolver=...)``
     - Instance
     - Recursively get all child space names
   * - ``to_dict()`` / ``from_dict(data)``
     - Instance/Class
     - Serialization and deserialization
   * - ``touch()``
     - Instance
     - Update timestamp

Hierarchy
^^^^^^^^^

Spaces support tree-like hierarchical structures, automatically managed by ``SemanticMapService`` via ``SpaceNamingPolicy``:

.. code-block::

   root
   ├── root_base_memory
   │   └── [unit_001, unit_002, ...]
   └── root_high_level_memory
       ├── root_episodic
       │   ├── root_episodic_summary
       │   └── root_episodic_event
       ├── root_knowledge
       │   ├── root_knowledge_summary
       │   └── root_knowledge_entity
       ├── root_emotional
       ├── root_procedural
       └── root_insights

.. _dsref-semantic-map:

SemanticMap
------------

``SemanticMapService`` (in ``mandol.application.semantic_map``) is the core service managing memory storage and vector indexing.
It integrates ``UnitStore`` (unit storage), ``AdaptiveVectorIndex`` (adaptive vector index), ``EmbeddingProvider`` (vectorization), and
``Reranker`` (reranking), providing complete infrastructure for semantic retrieval.

.. note::

   **Naming note**: The class is named ``SemanticMapService`` in code, accessed via the ``system.semantic_map`` property.
   Documentation uniformly abbreviates it as SemanticMap.

.. _dsref-semantic-graph:

SemanticGraph
---------------

``SemanticGraphService`` (in ``mandol.application.semantic_graph``) is the core service managing graph relationship structures between memory units.
It consists of explicit relationships (entity relations, event causality, traceability, coreference, etc.) and implicit semantic relationships (vector similarity),
with graph storage delegated to ``GraphStore`` (default: ``InMemoryGraphStore``).

.. note::

   **Naming note**: The class is named ``SemanticGraphService`` in code, accessed via the ``system.graph`` property.
   Documentation uniformly abbreviates it as SemanticGraph.

Core Interfaces
^^^^^^^^^^^^^^^

**Relationship Management**:

.. list-table::
   :header-rows: 1
   :widths: 55 45

   * - Method
     - Description
   * - ``add_relationship(source_uid, target_uid, relationship_name, **properties)``
     - Add relationship edge
   * - ``get_relationship(source_uid, target_uid, relationship_name) -> Optional[Dict]``
     - Query relationship edge properties
   * - ``delete_relationship(source_uid, target_uid, relationship_name=None)``
     - Delete relationship edge

**Graph Retrieval**:

.. list-table::
   :header-rows: 1
   :widths: 55 45

   * - Method
     - Description
   * - ``get_explicit_neighbors(uids, *, rel_type=None, direction="out") -> List[MemoryUnit]``
     - Get explicit relationship neighbors
   * - ``get_implicit_neighbors(uids, *, top_k=10) -> List[Tuple[MemoryUnit, float]]``
     - Get implicit semantic neighbors
   * - ``bfs_expand_units(seeds, *, per_seed=3, hops=1, rel_type=None) -> List[MemoryUnit]``
     - BFS graph expansion

Graph Structure Composition
^^^^^^^^^^^^^^^^^^^^^^^^^^^

SemanticGraph consists of two parts:

**Explicit Relationships**:

.. list-table::
   :header-rows: 1
   :widths: 18 40 42

   * - Edge Type
     - Description
     - Subtypes
   * - ``RELATED_TO``
     - Entity relationship edge
     - hometown, lives_in, works_at, located_in, part_of
   * - ``CAUSES``
     - Event causal relationship (A causes B)
     -
   * - ``CAUSED_BY``
     - Event causal relationship (B is caused by A)
     -
   * - ``INVOLVES``
     - Event-entity edge
     - participant, location, organizer, victim
   * - ``EVIDENCED_BY``
     - Traceability edge (points to raw conversation unit)
     -
   * - ``COREF``
     - Coreference edge (cross-session merging of same-referent entities/events)
     -
   * - ``ALIAS_OF``
     - Alias edge (entity alias relationship)
     -
   * - ``PRECEDES``
     - Temporal edge (chronological order of conversations/events)
     -
   * - ``FOLLOWS``
     - Temporal edge (chronological order of conversations/events)
     -

**Implicit Semantic Relationships**:

- ``SEMANTIC_SIMILAR`` edges built based on vector similarity
- Incrementally built during ``add()`` via ``_build_immediate_similarity_edges()``
- Edge weight is vector cosine similarity

Type System
------------

The system uses the following type aliases for type safety, all defined in ``mandol/domain/types.py``:

.. list-table::
   :header-rows: 1
   :widths: 20 30 50

   * - Type
     - Definition
     - Description
   * - ``Uid``
     - NewType
     - Memory unit unique identifier
   * - ``SpaceName``
     - NewType
     - Space name
   * - ``Embedding``
     - Type alias
     - Vector representation (typically float32 numpy array)

``SearchHit`` is defined in ``mandol/retrieval/types.py``:

.. code-block:: python

   @dataclass
   class SearchHit:
       unit: MemoryUnit
       final_score: float
       scores: Dict[str, float]
       ranks: Dict[str, int]
