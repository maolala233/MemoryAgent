项目简介与设计理念
==================

.. note::

   基础用户请参阅 :doc:`/basic-user/what-is-mandol`。本文档将在后续版本中重构，请更新你的书签。

项目背景
--------

AI Agent 在与用户交互过程中，需要像人类一样记忆、组织和检索对话历史。传统的记忆系统通常采用简单的向量数据库或键值存储，缺乏对记忆之间语义关系的建模，也无法支持多维度的记忆组织与检索。

Mandol 应运而生，它提供了一套完整的记忆管理解决方案，包括：

- **自动会话分割**：基于 LLM 的会话边界检测，将连续对话分割为有意义的会话单元
- **多维度记忆构建**：从原始对话中提取摘要、实体、事件、洞察等多类型记忆
- **语义图关系建模**：建立实体关系、事件因果、语义相似等多种关联
- **混合检索策略**：结合稠密向量、稀疏向量、关键词检索，并通过 RRF 融合与 BFS 扩展提升召回率

设计理念
--------

端口与适配器架构（六边形架构）
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

系统采用严格的端口与适配器模式分层：

- **domain/**：核心领域模型（MemoryUnit、MemorySpace、类型定义），不依赖任何外部框架
- **ports/**：抽象接口定义（EmbeddingProvider、LLMProvider、VectorIndex、GraphStore 等）
- **application/**：应用服务层（MemorySystem、SemanticMapService、SemanticGraphService 等），编排业务流程
- **infrastructure/**：基础设施实现（内存存储、FAISS、OpenAI 兼容的 LLM/Embedding 等）

这种架构使得系统可以灵活替换底层实现，例如：
- 将内存向量索引替换为 FAISS 索引
- 将内存图存储替换为 Neo4j
- 将本地 SentenceTransformers 替换为 OpenAI API

多视角记忆组织
^^^^^^^^^^^^^^^^

记忆不是扁平的列表，而是按语义维度组织的层级空间结构。每个会话会生成以下空间层级：

- **base_memory_{suffix}**：原始记忆单元
- **episodic_{suffix}**：情景记忆（摘要 + 事件）
- **knowledge_{suffix}**：知识记忆（摘要 + 实体）
- **emotional_{suffix}**：情感记忆
- **procedural_{suffix}**：程序记忆
- **insights_{suffix}**：洞察记忆

其中 ``{suffix}`` 是基于会话起始消息索引生成的唯一标识（如 ``msg_0_25``）。

这种组织方式支持细粒度的空间检索，例如"仅检索知识实体"或"仅检索事件"。

统一检索接口
^^^^^^^^^^^^^^

用户只需调用 ``system.holistic_retrieve(query, top_k)`` 即可完成全记忆检索。系统内部自动执行以下流程：

1. **分组召回**：将检索请求分发到 BASE、ENTITY、EVENT、SUMMARY 四个检索组
2. **三路召回**：每组内部执行 Dense + BM25 + Sparse 三路独立检索
3. **RRF 融合**：使用倒数排名融合方法合并多路结果
4. **BFS 扩展**：基于图关系扩展候选集
5. **全局重排**：所有候选合并后通过 Cross-Encoder Reranker 重排序

核心能力
--------

- **会话管理**：自动检测会话边界，支持手动和自动两种会话分割模式
- **多类型记忆提取**：支持摘要、实体、事件、洞察、全局洞察五种记忆类型
- **关系建模**：实体关系、事件因果、语义相似、证据关联
- **跨会话合并**：自动合并不同会话中相同指代的实体/事件
- **持久化**：支持 JSON 格式的完整状态导出与加载
- **混合检索**：三路召回 + RRF + BFS + Reranker 的统一检索策略
