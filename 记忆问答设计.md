
# Codex Memory 系统改造 - 从 CLI 到 Web 全栈的模块化设计

## 📋 项目概述

**原始项目**: https://github.com/mcncarl/codex-memory

**当前状态**: 
- Codex Agent 的长期记忆系统
- 纯 CLI 工具集 + Markdown 记忆库 + SQLite 索引
- 已有: 关键词搜索、向量检索、Agent 进化追踪

**改造需求**:
1. 解耦 Codex 强绑定 → 支持多 LLM 配置化
2. 从 CLI 升级到 Web UI
3. 支持多智能体管理和编排
4. 保持现有 Python 脚本和 Markdown 结构

---

## 🏗️ 架构改造方案

### 分层设计
```
客户端层 (Web UI)
↓ API 层 (FastAPI)
↓ 业务服务层 (Service)
↓ 适配器层 (LLM、Agent、存储) 
↓ 数据层 (SQLite、向量库、文件系统)
```


## 📦 后端服务模块设计

### 模块 1: LLM 适配器层 (`backend/services/llm_adapter.py`)

**职责**: 抽象 LLM 调用，支持多个提供商

**核心方法**:
- `LLMProvider` (基类)
  - `async generate(prompt, system_prompt, temperature, max_tokens)` → 同步生成
  - `async stream_generate(prompt, system_prompt)` → 流式生成
  - `validate_connection()` → 验证连接

- `OllamaProvider` (本地模型)
  - 实现 generate 和 stream_generate
  - 支持模型列表获取
  - 处理 timeout 和重试

- `OpenAIProvider` (云端 API)
  - 实现 generate 和 stream_generate
  - Token 计数
  - 速率限制处理

- `LLMFactory`
  - `create(config: dict)` → 根据配置返回对应 Provider 实例
  - `register_provider(name, provider_class)` → 动态注册新提供商

**配置来源**: `config/models.yaml`

---

### 模块 2: Agent 管理器 (`backend/services/agent_service.py`)

**职责**: Agent 定义、注册、运行时管理

**核心方法**:
- `AgentRegistry`
  - `load_config(config_path)` → 从 YAML 加载 Agent 定义
  - `get_agent(agent_name)` → 获取或创建 Agent 实例
  - `list_agents()` → 列出所有可用 Agent
  - `validate_agent_config(config)` → 验证 Agent 配置合法性

- `Agent` (单个 Agent 实例)
  - `async run(task, context)` → 执行任务
  - `async stream_run(task, context)` → 流式执行任务
  - `_retrieve_memories(task)` → 根据策略检索记忆
  - `_build_prompt(task, memories, context)` → 构建完整提示词
  - `_call_tools(tool_name, args)` → 调用工具
  - `_parse_response(response)` → 解析 LLM 输出

- `Tool` (工具基类)
  - `search_memory(query, filters)` → 搜索记忆库
  - `write_memory(path, content)` → 写入记忆文件
  - `execute_code(code)` → 执行代码片段
  - `read_memory(path)` → 读取记忆文件

**配置来源**: `config/agents.yaml`

---

### 模块 3: 记忆检索服务 (`backend/services/retrieval_service.py`)

**职责**: 统一记忆检索接口，支持混合策略

**核心方法**:
- `MemoryRetriever`
  - `keyword_search(query, limit, filters)` → SQLite FTS5 关键词搜索
  - `semantic_search(query, limit, filters)` → Embedding + 向量检索
  - `hybrid_search(query, limit, filters)` → 关键词 + 语义混合
  - `none_search()` → 返回空结果
  - `get_document(rel_path)` → 获取单个文档完整内容

- `SearchStrategy` (枚举)
  - KEYWORD (纯关键词)
  - SEMANTIC (纯语义)
  - HYBRID (混合，先关键词后语义)
  - NONE (不搜索)

**实现细节**:
- 混合搜索的合并策略（权重、去重、排序）
- 缓存热点搜索结果
- 过滤逻辑（track、memory_type、project_id 等）

---

### 模块 4: 记忆库服务 (`backend/services/memory_service.py`)

**职责**: 记忆库的 CRUD、索引、统计

**核心方法**:
- `MemoryService`
  - `get_document(rel_path)` → 返回单个文件元数据 + 内容
  - `list_documents(filters)` → 列表查询，支持分页和过滤
  - `create_document(path, content, metadata)` → 新建文件
  - `update_document(path, content)` → 编辑文件
  - `delete_document(path)` → 删除文件（标记为 deleted）
  - `rescan_vault()` → 全库重新索引
  - `get_stats()` → 统计信息（文档数、类型分布、大小）
  - `get_open_loops()` → 获取所有未闭环项
  - `watch_changes()` → 监听文件系统变化（WebSocket 用）

**返回类型**:
- `MemoryDoc` (数据模型)
  - path, rel_path, title, content
  - metadata (memory_type, track, project_id, status 等)
  - summary, next_hint, open_loop_count
  - indexed_at, verified_at

---

### 模块 5: 配置加载器 (`backend/services/config_loader.py`)

**职责**: 配置加载、验证、环境变量覆盖

**核心方法**:
- `ConfigLoader`
  - `load_models_config(path)` → 加载 models.yaml
  - `load_agents_config(path)` → 加载 agents.yaml
  - `load_retrieval_config(path)` → 加载 retrieval.yaml
  - `apply_env_overrides(config)` → 环境变量覆盖配置
  - `validate_config(config, schema)` → 验证配置合法性

**配置优先级**:
1. 配置文件默认值
2. 环境变量覆盖
3. 运行时参数覆盖

## 🔌 API 路由设计

### 模块 6: 搜索路由 (`backend/routers/search.py`)

**职责**: 暴露搜索功能为 HTTP API

**端点**:
- `POST /api/search` (搜索记忆)
  - 入参: query, limit, track, memory_type, use_semantic
  - 出参: List[MemoryResult]
  - 功能: 调用 `MemoryRetriever.hybrid_search()` 或 `keyword_search()`

- `GET /api/search/suggestions` (搜索建议)
  - 入参: q (查询前缀), limit
  - 出参: List[str]
  - 功能: 从 SQLite FTS 索引获取建议

- `GET /api/search/filters` (获取可用过滤项)
  - 出参: {tracks: [], memory_types: [], projects: []}
  - 功能: 从 memory_docs 表获取不同的字段值

---

### 模块 7: 记忆库路由 (`backend/routers/memory.py`)

**职责**: 记忆文件的 CRUD API

**端点**:
- `GET /api/memory/{doc_path}` (获取单个文件)
  - 出参: MemoryDocResponse
  - 功能: 读取 Markdown 文件，返回元数据 + 内容

- `PUT /api/memory/{doc_path}` (编辑文件)
  - 入参: content, metadata (可选)
  - 出参: {status: "updated", path, version}
  - 功能: 更新文件，触发重新索引

- `POST /api/memory` (新建文件)
  - 入参: rel_path, content, metadata
  - 出参: {status: "created", path}
  - 功能: 创建新 Markdown 文件，设置 frontmatter

- `DELETE /api/memory/{doc_path}` (删除文件)
  - 出参: {status: "deleted"}
  - 功能: 标记文件为 deleted (逻辑删除)

- `GET /api/memory` (列表查询)
  - 入参: skip, limit, track, memory_type, status, has_open_loop
  - 出参: {total, items: List[MemoryDoc]}
  - 功能: 分页列表，支持多维过滤

- `POST /api/memory/rescan` (重新索引)
  - 出参: {docs_indexed, duration_seconds}
  - 功能: 扫描所有 MD 文件，更新 SQLite 索引

- `GET /api/memory/stats` (统计信息)
  - 出参: {total_docs, total_size, by_type: {}, by_track: {}, open_loops_count}
  - 功能: 返回统计数据

---

### 模块 8: 聊天路由 (`backend/routers/chat.py`)

**职责**: Agent 交互的同步和流式 API

**端点**:
- `POST /api/chat` (同步对话)
  - 入参: message, agent, context (可选)
  - 出参: {response, memories_used: [], thinking, status}
  - 功能: 调用 `AgentService.run_agent()`，等待完成

- `WebSocket /api/chat/stream/{agent_id}` (流式对话)
  - 接收消息: {message, context}
  - 发送流: {type: "chunk"|"done"|"memories", content}
  - 功能: 实时流式返回 Agent 输出

---

### 模块 9: Agent 管理路由 (`backend/routers/agents.py`)

**职责**: Agent 定义和状态管理

**端点**:
- `GET /api/agents` (列表所有 Agent)
  - 出参: List[{name, role, capabilities, memory_strategy}]

- `GET /api/agents/{agent_id}` (获取 Agent 详情)
  - 出参: {name, role, capabilities, tools, dependencies, llm_model}

- `POST /api/agents/{agent_id}/test` (测试 Agent)
  - 入参: test_prompt
  - 出参: {response, latency_ms, status}
  - 功能: 测试 Agent 和 LLM 连接

---

### 模块 10: 统计路由 (`backend/routers/stats.py`)

**职责**: 仪表板数据

**端点**:
- `GET /api/stats/overview` (总览)
  - 出参: {total_docs, total_size, last_updated}

- `GET /api/stats/distribution` (分布)
  - 出参: {by_type: {}, by_track: {}, by_status: {}}

- `GET /api/stats/timeline` (更新时间线)
  - 入参: days (7|30|90)
  - 出参: List[{date, doc_count, update_count}]

- `GET /api/stats/open-loops` (开放问题)
  - 出参: List[{path, title, kind, item, count}]

## 模块 28: 文档上传和解析服务

### 整体流程
```
用户上传: project-plan-2026.pdf
    ↓
后端解析:
    ├─ 识别出 5 个章节
    ├─ 提取 200+ 句子
    ├─ 分成 15 个语义块
    └─ 元数据: {title: "项目规划", author: "PM", date: 2026-01-01}
    ↓
转换为记忆:
    ├─ 块 1 → 项目/2026-规划-概述.md (memory_type: project)
    ├─ 块 2 → 工作流/2026-规划-流程.md (memory_type: workflow)
    ├─ 块 3 → 决策/2026-规划-权衡.md (memory_type: decision)
    └─ ...
    ↓
LLM 处理每个块:
    ├─ 生成 summary (frontmatter 用)
    ├─ 提取 keywords
    ├─ 识别 open_loops (行动项)
    └─ 推荐 memory_type 和 track
    ↓
用户预览 + 编辑:
    ├─ 修改某些文件的 frontmatter
    ├─ 删除不相关的块
    └─ 调整 project_id 分类
    ↓
保存到记忆库:
    ├─ 创建 15 个 .md 文件
    ├─ 写入 SQLite 索引
    ├─ 生成向量 embedding
    └─ WebSocket 通知前端
    ↓
搜索验证:
    用户搜索 "2026 规划"
```

**文档解析服务 (`backend/services/document_parser.py`)**
职责: 从各种格式文档提取内容
核心方法:
- `DocumentParser`
  - `parse_pdf(file_path)` → {text, metadata, pages}
    - 使用: PyPDF2/pdfplumber
    - 提取: 文本 + 图片 + 元数据 + 页码
  
  - `parse_docx(file_path)` → {text, metadata, sections}
    - 使用: python-docx
    - 提取: 段落 + 表格 + 样式信息
  
  - `parse_markdown(file_path)` → {text, metadata, structure}
    - 提取: 标题层级 + 代码块 + 链接
  
  - `parse_text(file_path)` → {text}
    - 纯文本解析
  
  - `extract_metadata(content)` → {title, author, date, tags}
    - 从内容推断元数据

**文档分块服务 (`backend/services/chunking_service.py`)**
职责: 将长文档分解成记忆片段
核心方法:
- `ChunkingService`
  - `chunk_by_section(text, headings)` → List[Chunk]
    - 按标题分块（优先）
    - 保留上下文
  - `chunk_by_size(text, max_tokens, overlap)` → List[Chunk]
    - 按字数/token 数分块
    - 支持重叠 (overlap) 保留连贯性
  - `chunk_by_semantic(text)` → List[Chunk]
    - 基于语义相似度分块 (可选，精细)
    - 使用 Embedding 模型
  - `merge_short_chunks(chunks, min_tokens)` → List[Chunk]
    - 合并过短的分块
数据结构:
- `Chunk`
  - text: str (块内容)
  - section: str (所属章节)
  - tokens: int (token 数)
  - index: int (块序号)

**文档到记忆转换服务 (`backend/services/doc_to_memory.py`)**
职责: 将文档块转换成结构化记忆文件
核心方法:
- `DocToMemoryConverter`
  - `convert_chunk_to_memory(chunk, original_metadata)` → MemoryFile
    - 为单个分块生成一个 Markdown 文件
    - 自动生成 frontmatter
  
  - `generate_frontmatter(chunk, doc_metadata)` → dict
    - memory_type: "imported_document"
    - track: 从内容推断（project/workflow/decision）
    - project_id: 从文件名或用户输入
    - keywords: 从 LLM 提取关键词
    - source_doc: 原文档路径
    - section: 来自哪个章节
  
  - `summarize_chunk(chunk)` → str
    - 调用 LLM 生成摘要 (作为 summary frontmatter)
  
  - `classify_chunk(chunk)` → (memory_type, track, keywords)
    - 调用 LLM 分类这个块属于什么类型的记忆
    - memory_type: project | workflow | decision | note
    - track: 对应的 track
    - keywords: 提取的关键词列表
  
  - `extract_action_items(chunk)` → List[str]
    - 从内容提取待办项（作为 open_loops）

数据结构:
- `MemoryFile`
  - filename: str (生成的文件名)
  - rel_path: str (相对路径，如 项目/xxx.md)
  - frontmatter: dict (YAML 元数据)
  - content: str (Markdown 内容)

**文档上传处理路由 (`backend/routers/documents.py`)**

职责: 暴露文档上传和处理的 API
端点:
- `POST /api/documents/upload` (上传文档)
  - 入参: file (multipart/form-data)
  - 出参: {file_id, filename, status, file_size, page_count}
  - 功能: 
    - 保存临时文件
    - 验证格式 (PDF/DOCX/MD/TXT)
    - 返回文件 ID 用于后续处理

- `POST /api/documents/{file_id}/parse` (解析文档)
  - 出参: {chunks: List[{text, section, index}], metadata, total_chunks}
  - 功能:
    - 调用 DocumentParser
    - 返回预览 (first 3 chunks)

- `POST /api/documents/{file_id}/convert-to-memory` (转换为记忆)
  - 入参: {project_id, memory_type, strategy: "auto"|"manual"}
    - strategy: 自动分类还是用户手动指定
  - 出参: {memory_files: List[{rel_path, content, frontmatter}]}
  - 功能:
    - 解析文档
    - 分块
    - 转换为记忆文件
    - 返回预览（用户可在保存前查看）

- `POST /api/documents/{file_id}/save` (保存到记忆库)
  - 入参: {memory_files: List[{rel_path, content, frontmatter}]}
    - 用户可编辑后再保存
  - 出参: {saved_count, paths: List[str]}
  - 功能:
    - 写入 Markdown 文件
    - 触发重新索引
    - 返回保存的文件路径

- `DELETE /api/documents/{file_id}` (删除临时文件)

- `GET /api/documents/status/{file_id}` (查询处理进度)
  - 出参: {status: "uploaded"|"parsing"|"converting"|"done", progress: 0-100}
  - 功能: 用于大文件处理时显示进度

## 📊 数据流和交互示例

### 场景 1: 用户搜索记忆
```
用户输入查询 
↓ 前端 SearchBox 组件获取建议 → /api/search/suggestions

↓ 返回 List[MemoryResult] 到前端 
↓ 前端 ResultCard 列表展示结果
```
### 场景 2: 用户与 Agent 对话
```
用户输入任务 + 选择 Agent 
↓ 前端 ChatBox 通过 WebSocket 连接到 /api/chat/stream/{agent_id} 
↓ 后端 ChatRouter 调用 AgentService.run_agent() 
↓ AgentService 调用 Agent.stream_run() 
↓ Agent 检索记忆 → MemoryRetriever.hybrid_search() 
↓ Agent 调用 LLM → LLMFactory.create().stream_generate() 
↓ LLM 流式返回内容，后端通过 WebSocket 发送到前端 
↓ 前端实时显示流式内容 + 记忆引用
```
### 场景 3: 用户编辑记忆文件
```
用户打开编辑器 → /memory/editor/[path] 
↓ 前端加载文件内容 → /api/memory/{path} 
↓ 用户编辑 Markdown 和 frontmatter 
↓ 前端自动保存 (防抖) → /api/memory/{path} (PUT) 
↓ 后端更新文件，触发重新索引 
↓ WebSocket 广播文件更新事件 
↓ 其他客户端收到更新通知
```

5. **错误恢复**: 网络中断时前端应有重连机制，后端应有幂等性保证
